## Sidebar 和 Navbar Dropdown 有什麼差別？

**Q：** 我的 homepage sidebar 是看不出資料夾分類的對嗎？他是用 `[xx|]` 區分嗎？

**A：** 分兩個地方要分開理解：

**Sidebar（左側欄，進入文章後才出現）**
由資料夾結構自動產生，`CodeShiba學習練習/`、`練習筆記/`、`Claude教學練習紀錄/` 各自是一個可折疊的分類，看得出層級。由 `_category_.json` 控制名稱與順序。

**Navbar Dropdown（頂部選單點開的清單）**
是手動寫在 `docusaurus.config.js` 的平面清單，沒有分組功能（beta.6 限制），所以用 `CodeShiba｜`、`練習｜`、`Claude教學｜` 前綴來視覺上區分。

從首頁點進去某一篇文章後，左側 sidebar 才會出現，那裡的分類是看得清楚的。Navbar 的平面清單只是入口捷徑。

## Docusaurus beta.6 是什麼？

**Q：** beta.6 是什麼？

**A：** 就是 Docusaurus 的版本號。

`beta.6` = **2.0.0-beta.6**，是 2021 年左右的測試版。目前最新是 **3.x 正式版**，差了很多個版本。

你的網站當初建的時候用的就是這個版本，所以有些新功能在這個舊版本裡沒有，用了就會報錯。

## type: 'html' 是什麼？為什麼會報錯？

**Q：** `type: 'html'` 是幹嘛的？為什麼在這個版本用了會報錯？

**A：** 在 Docusaurus 新版裡，`type: 'html'` 讓你在 navbar dropdown 裡插入**任意 HTML**，用來做分隔線或小標題，達到分組效果：

但因為這個功能是**後來才加進去**的，beta.6 的時候還不存在。Docusaurus 讀到 `type: 'html'`，查自己支援的類型清單，找不到就報錯。

類比：叫一支 iOS 12 的手機跑只有 iOS 16 才有的功能，手機不認識，直接閃退。

解法：改回用 label 前綴（`CodeShiba｜`、`練習｜`）來視覺區分，這在所有版本都支援。

## YAML frontmatter 的 id 為什麼要加引號？

**知識點：** Docusaurus MD 檔的 frontmatter `id` 若是日期格式，不加引號會出錯。

**原因：** YAML 會把 `2026-05-06` 自動解讀為日期物件（Date），Docusaurus 再把它轉成字串時變成 `Wed May 06 2026 08:00:00 GMT+0800`，這個字串變成 URL 路徑就完全爛掉，導致 broken links 報錯。

**正確寫法：**

```yaml
id: "2026-05-06"
```

## Docusaurus 資料夾名稱為什麼不能用中文？

**知識點：** `docs/` 下的資料夾名稱若含中文，會導致 URL 不一致。

**原因：** 中文字元在 URL 裡需要 percent-encoding（如 `問` → `%E5%95%8F`）。Docusaurus 產生的實際頁面路徑是編碼後的 URL，但 navbar 的 `to` 若寫中文，兩者對不上，就出現 broken links。

**解法：** 資料夾名稱一律用英文，例如 `ClaudeQA/` 而不是 `Claude問答/`。

## Docusaurus 的 .md 和 .mdx 有什麼差別？

**Q：** 加密元件為什麼要改成 `.mdx`？`.md` 不是也可以嗎？

**A：** 你是對的，`.md` 也可以。

這個版本（Docusaurus beta.6）對 `.md` 檔案也支援 JSX 語法，不需要特別改成 `.mdx`。我誤用了舊筆記裡「需使用 MDX 非 .md」的說法，但實際 build 測試證明 `.md` 完全沒問題。

`.mdx` 是明確宣告「這是 MDX 檔」的副檔名，語意更清楚，但在這個版本裡不是必要的。

## 為什麼 IG API 一定要有 Facebook 粉專？

**Q：** 我只是要發 IG 貼文，為什麼還要扯到 Facebook 粉專？

**A：** 這是 Meta 的 API 規定。Meta 把 Facebook 和 Instagram 的權限管理統一放在 Facebook 這一側，IG 帳號必須「掛在」一個 Facebook 粉專底下，程式才能透過粉專取得操控 IG 的授權。

粉專本身不需要真的經營，只是一個技術橋梁。就像你要用某個 App 登入，它要求你必須有 Google 帳號才能授權——Google 帳號不是用來操作 App，只是身份驗證的入口。

## 什麼是 Access Token？為什麼不直接用帳號密碼？

**Q：** 為什麼程式不直接用帳號密碼登入 IG，而是要用 Token？

**A：** 安全性考量。直接給程式帳號密碼有幾個風險：

- 程式碼一旦外洩，帳號就被拿走了
- 程式拿到密碼後可以做任何事（改密碼、刪帳號）
- 你沒辦法限制它只能做特定操作

**Access Token 的設計是：**
- 只給程式一張「授權票」，不給密碼
- 票上寫清楚「只能發文，不能改密碼」等權限範圍
- 票有有效期限，過期就失效
- 票可以隨時撤銷，不影響帳號本身

這套機制叫 **OAuth 2.0**，是現代 API 的標準做法。

## 什麼是 API？

**Q：** API 到底是什麼？

**A：** API 是「服務的遙控器」。

以 Instagram 為例：Meta 在背後架了一套系統，開了一個「窗口」，讓外部程式按照規定的格式發送請求——「我要幫這個帳號發一張圖片，說明文字是這個」——Meta 的系統收到後就去執行，貼文就出現在 IG。

你的 Python 腳本不是真的去打開 IG 網頁、移動滑鼠點發文，而是直接用 API 這個窗口下指令。更快、更穩定，也是自動化的基礎。

## 為什麼 IG API 發文一定要有圖片？

**Q：** 為什麼不能純文字發文？

**A：** Instagram 本身就不支援純文字貼文，這是平台設計，不是 API 的限制。IG 的設計定位是圖像優先的平台，每一篇貼文都必須附上圖片或影片。

透過 API 發文時，文字內容是作為圖片的「說明（caption）」存在，不是獨立的文字貼文。

## 為什麼個人 IG 帳號不能用 API？

**Q：** 切換成商業帳號之前，為什麼不能用？

**A：** Meta 把 Instagram API 的使用限制在商業帳號（Business）或創作者帳號（Creator），個人帳號無法存取。

原因是 API 的設計目的是給「品牌或內容創作者」用來做行銷和自動化管理，Meta 不開放給一般個人用戶，避免被拿來大量自動化操作或濫用。切換到商業帳號就等於告訴 Meta「我是專業用途」，才開放 API 使用權限。

## .env 檔是什麼？為什麼不把密鑰寫在程式裡？

**Q：** 為什麼要另外建一個 .env 檔？直接寫在 Python 檔裡不行嗎？

**A：** 可以執行，但不安全。

如果把 App Secret 和 Token 寫死在 `ig_post.py` 裡，只要這個檔案被分享出去（傳給朋友看、上傳到 GitHub），別人就拿到你的憑證，可以用你的身份對 Instagram 做任何事。

`.env` 的設計是：
- 憑證放在獨立的隱藏檔案（`.` 開頭代表隱藏）
- 程式碼只寫「去 .env 裡讀取」，不寫實際的值
- `.env` 不進版本控制（加到 `.gitignore`），只存在本機

這樣即使程式碼公開，別人也看不到你的密鑰。

## Instagram Token 和 Facebook User Token 有什麼不同？

**Q：** 我拿到的 Token 為什麼不能用 `me/accounts`？

**A：** Token 分好幾種，每種能呼叫的 API 不同：

| Token 類型 | 說明 | 能呼叫的 API |
|------------|------|-------------|
| Facebook User Token | 代表 Facebook 用戶身份 | `me/accounts`（取得粉專列表）等 Facebook API |
| Instagram Token | 代表 Instagram 帳號身份 | IG 相關 API（發文、取得貼文）|
| Page Access Token | 代表粉專身份 | 粉專相關 API |

今天拿到的是 Instagram Token（透過 Instagram Login 流程產生），所以打 `me/accounts`（Facebook API）會報錯——就像拿台灣的駕照去開美國的車，類型對不上。

解法：直接用 Instagram Token 打 IG API，不需要繞 Facebook Page 那條路。

## 憑證在對話裡輸入過，要怎麼處理？

**Q：** 我把 App Secret 直接輸入在 Claude 對話裡了，這樣有問題嗎？

**A：** 有風險，建議立刻重新產生。

對話內容會存在 Anthropic 的伺服器上，雖然不是公開的，但原則上你不應該把任何密鑰留在對話記錄裡。

**處置方式：**
1. 去 Meta Developer → 應用程式設定 → 重設應用程式密鑰 → 產生新的 App Secret
2. 舊的 App Secret 立刻作廢，任何拿到舊 Secret 的人都無法使用
3. 更新本機 `.env` 裡的 `APP_SECRET` 為新值

這個動作叫「憑證輪替（Credential Rotation）」，是安全事件的標準處置。不管是密碼外洩、Token 被截取，還是不小心把密鑰印在對話、截圖、文件裡，都應該立刻輪替，讓舊的作廢。
